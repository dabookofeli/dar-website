Formal definition

Receipts are evidence, not telemetry.

Digital Action Receipts (DARs) are durable, append-only proof that a specific action occurred—issued at action time, preserved under customer custody, and independently verifiable later.

Definition

DAR = a verifiable, append-only record of a digital action, issued at execution time. It preserves: • actor • action • object / target • timestamp • stable reference (receipt_ref) • policy / context references (optional) • integrity proof (signature / hash chain)

Why this exists

When systems are questioned later, logs and narratives drift. DAR provides a stable reference that survives remediation and hindsight.

disputes audits incidents claims investigations

What DAR is

Evidence artifact

DAR records facts: what happened, when, and under what referenced context.

Issued at commit time

Receipts are emitted after an action commits (non-blocking), not reconstructed later.

Independently verifiable

Unlike logs, receipts provide tamper-evident proof that can be verified outside the system.

Customer custody

Receipts are stored in customer-controlled, immutable/append-only storage.

What DAR is not

Not monitoring

No surveillance. No dashboards required. No continuous observation layer.

Not enforcement

DAR does not decide, block, score, or interpret runtime behavior.

Not a system of record

DAR preserves evidence. Source systems keep authority over data and state.

Not raw data storage

Receipts store references (policy_version_ref, model_version_hash, config_ref), not sensitive payloads.

Receipts store references (policy versions, model hashes, config refs)—not raw data.

No surveillance, no monitoring, no decision logic.

Philosophy

In disputes, audits, and investigations, the system is not the witness. The receipt is.
DAR does not replace systems of record. DAR replaces reconstruction—by giving the customer an evidence artifact that stands independently of the vendor, the platform, or the operator.

Custody & storage

Receipts are written into customer-owned infrastructure. DAR does not act as a system of record.

  • AWS S3 Object Lock
  • Azure Immutable Blob
  • GCP Retention Bucket

Minimal receipt example

{ "id": "8F6A92", "ts": "2026-01-04T15:24:11Z", "actor": "agent|user|system", "action": "transaction.executed", "object": "order-123", "receipt_ref": "internal-id-123", "policy_version_ref": "policy-v7", "model_version_hash": "sha256:…", "integrity": { "signature": "…", "hash_chain_prev": "…" } }

This is intentionally compact: enough to prove the action occurred and bind it to referenced context.
