Boundaries & invariants

Principles

DAR is intentionally narrow. The product is evidence—issued at action time, held under customer custody, and verifiable later. These principles protect that boundary.

Core framing
In disputes, audits, and investigations, the system is not the witness. The receipt is.

Invariants (must always be true)

  • Issued at action time — receipts are emitted after an action commits (non-blocking).
  • Append-only — receipts are not editable; corrections are new receipts.
  • Customer custody — receipts are stored in customer-controlled infrastructure.
  • Independent verification — integrity can be checked outside the issuing system.
  • Minimal payload — store references (versions/hashes/refs), not raw data.
  • Portable evidence — receipts remain meaningful across migrations and vendors.

Non-goals (what DAR will not become)

  • Not monitoring — no surveillance layer, no always-on observation.
  • Not decision logic — DAR does not approve/deny, score, or interpret outcomes.
  • Not enforcement — DAR does not block actions or act as a gatekeeper.
  • Not a system of record — source systems remain authoritative for data and state.
  • Not a data lake — receipts don’t store sensitive payloads by default.

Evidence over narrative

Narratives change. Logs are internal. DAR is an externalizable artifact that persists facts.

Question later: "What happened?" DAR answers with a receipt + integrity proof + context references: • actor · action · object · time • policy_version_ref / model_version_hash / config_ref • signature / hash chain pointers

Reference, don’t retain

DAR preserves accountability without becoming a repository of private data.

  • Use policy_version_ref instead of storing policy text
  • Use model_version_hash instead of prompts/training data
  • Use config_ref instead of full configs
  • Store payloads only when explicitly required and governed

What “customer custody” means

Receipts are written into customer-controlled immutable storage. DAR can provide issuance/verification tooling, but does not become the owner of the evidence.

  • AWS S3 Object Lock
  • Azure Immutable Blob
  • GCP Retention Bucket

This prevents a core failure mode: the vendor controlling the authoritative evidence during a dispute.
